A Proof of Origin token (commonly abbreviated `po_token`) is a short-lived attestation token generated by YouTube's BotGuard/DroidGuard system. When YouTube requires attestation, it issues a challenge that a legitimate client must solve using a JavaScript-based virtual machine embedded in the page. The resulting token proves to YouTube that the session originates from an authentic client environment. Without a valid token, requests for streaming URLs may be throttled or return unusable format data.

<Note>
  A PO token is not required in all cases. For many use cases — reading metadata, searching, browsing channels — no token is needed. Tokens are most relevant when accessing high-quality streaming formats or when YouTube has detected unusual request patterns.
</Note>

## What the token represents

The token is passed to InnerTube inside the `serviceIntegrityDimensions` context object of a player request:

```json
{
  "serviceIntegrityDimensions": {
    "poToken": "<your-token-here>"
  }
}
```

YouTube.js handles this injection automatically. You supply the raw token string; the library wraps it in the correct structure before every player request.

## Passing a token at session level

Set `po_token` in `Innertube.create()` to bind the token to the entire session. Every `getInfo()` and `getBasicInfo()` call made through this instance will include the token unless overridden per-request.

```ts
import { Innertube, UniversalCache } from 'youtubei.js';

const yt = await Innertube.create({
  po_token: 'YOUR_PO_TOKEN',
  visitor_data: 'YOUR_VISITOR_DATA',
  cache: new UniversalCache(false),
  generate_session_locally: true,
});

const video = await yt.getInfo('dQw4w9WgXcQ');
console.log(video.basic_info.title);
```

The session stores the token on `session.po_token`. You can read it back at any time:

```ts
console.log(yt.session.po_token);
```

## Passing a token per request

`getInfo()` and `getBasicInfo()` both accept a `GetVideoInfoOptions` object with an optional `po_token` field. A per-request token takes precedence over the session-level token.

```ts
import { Innertube } from 'youtubei.js';

const yt = await Innertube.create();

const video = await yt.getInfo('dQw4w9WgXcQ', {
  po_token: 'VIDEO_SPECIFIC_TOKEN',
});
```

This is useful when a token is bound to the specific video ID being requested, rather than to the session as a whole.

### `GetVideoInfoOptions` fields

<ResponseField name="client" type="InnerTubeClient" optional>
  Override the InnerTube client used for this request (e.g. `'WEB'`, `'ANDROID'`, `'IOS'`).
</ResponseField>

<ResponseField name="po_token" type="string" optional>
  Proof of Origin token bound to this video ID. If omitted, the session-level `po_token` is used. If neither is set, no token is sent.
</ResponseField>

## How the token is applied internally

In `Innertube.ts`, every player request checks for a token in this order:

1. `options.po_token` (per-request)
2. `session.po_token` (session-level)
3. No token — the `serviceIntegrityDimensions` field is omitted entirely

```ts
// From src/Innertube.ts
if (options?.po_token) {
  extra_payload.serviceIntegrityDimensions = {
    poToken: options.po_token
  };
} else if (session.po_token) {
  extra_payload.serviceIntegrityDimensions = {
    poToken: session.po_token
  };
}
```

This applies identically to both `getInfo()` (which fetches player and watch-next responses) and `getBasicInfo()` (which fetches the player response only).

## Pairing po_token with visitor_data

The token is cryptographically bound to a visitor ID. You must pass the same `visitor_data` string that was used when generating the token. Mismatched visitor data will produce an invalid attestation.

```ts
import { Innertube, ProtoUtils, Utils } from 'youtubei.js';

// Generate a consistent visitor data string
const visitorData = ProtoUtils.encodeVisitorData(
  Utils.generateRandomString(11),
  Math.floor(Date.now() / 1000)
);

const yt = await Innertube.create({
  po_token: await getTokenForVisitor(visitorData), // your token generation logic
  visitor_data: visitorData,
  generate_session_locally: true,
});
```

<Tip>
  `ProtoUtils.encodeVisitorData()` produces a base64-encoded protobuf that YouTube expects as the `X-Goog-Visitor-Id` header value. It encodes a visitor identifier string and a Unix timestamp.
</Tip>

## Generating a token in the browser

In browser environments, tokens can be generated using the `bgutils-js` library, which implements the BotGuard challenge/response flow. The browser example in the repository at `examples/browser/web/src/main.ts` demonstrates the full flow:

```ts
import { BG } from 'bgutils-js';
import { Innertube, ProtoUtils, Utils, UniversalCache } from 'youtubei.js/web';

const visitorData = ProtoUtils.encodeVisitorData(
  Utils.generateRandomString(11),
  Math.floor(Date.now() / 1000)
);

// Mint a placeholder token immediately for cold-start playback
const coldStartToken = BG.PoToken.generatePlaceholder(visitorData);

// Resolve the real token in the background
let poToken: string | undefined;
getWebPoToken(visitorData).then((token) => { poToken = token; });

const yt = await Innertube.create({
  po_token: poToken || coldStartToken,
  visitor_data: visitorData,
  fetch: yourProxyFetchFunction,
  generate_session_locally: true,
  cache: new UniversalCache(false),
});

async function getWebPoToken(identifier: string): Promise<string | undefined> {
  const requestKey = 'O43z0dpjhgX20SCx4KAo';
  const bgConfig = { fetch, globalObj: window, requestKey, identifier };

  const bgChallenge = await BG.Challenge.create(bgConfig);
  if (!bgChallenge) throw new Error('Could not get challenge');

  const interpreterJs =
    bgChallenge.interpreterJavascript.privateDoNotAccessOrElseSafeScriptWrappedValue;

  if (interpreterJs) {
    new Function(interpreterJs)();
  } else {
    throw new Error('Could not load VM');
  }

  const result = await BG.PoToken.generate({
    program: bgChallenge.program,
    globalName: bgChallenge.globalName,
    bgConfig,
  });

  return result.poToken;
}
```

<Warning>
  The `bgutils-js` approach requires executing JavaScript retrieved from Google's servers. Only use it in environments where that is acceptable and where you control the execution context.
</Warning>

## When po_token is required vs optional

| Scenario | Token required? |
|---|---|
| Reading video metadata (`basic_info`, `primary_info`) | No |
| Browsing channels, playlists, home feed | No |
| Searching | No |
| Accessing deciphered streaming URLs (standard quality) | Sometimes |
| Accessing high-quality or age-restricted streaming formats | Likely yes |
| Requests that trigger YouTube's bot detection | Yes |
| Authenticated requests with cookies | Usually not required |

In practice, whether a token is required depends on the client type, the request frequency, the IP address reputation, and YouTube's continuously updated detection heuristics. Start without a token and add one if you observe throttled bitrates, missing formats, or playability errors.
